## V2.3 — Researcher Portability Layer

Three features that belong together as a cluster. Goal: make the system
usable by researchers and narrative designers who are not the developers.

### Externalizable Vocabulary Collections

The word lists in `mccf_voice_api.py` are currently hardcoded and calibrated
for Ollama constitutional language. V2.3 target: extract to external JSON files
per domain:

```
vocabulary/
  default.json          ← current Ollama constitutional vocabulary (V2.3 expanded)
  clinical.json         ← therapeutic/medical domain
  legal.json            ← negotiation/adversarial domain
  educational.json      ← learning/assessment domain
  narrative.json        ← creative/story domain
```

Each file defines pos_words, neg_words, uncertainty_words, and per-channel
decomposition vocabulary. `mccf_voice_api.py` loads the active vocabulary
on startup from a config setting. Researchers swap vocabulary files without
touching Python code.

### Waypoint Editor with Domain-Specific Question Groups

The constitutional arc waypoints and questions are currently hardcoded in
`mccf_constitutional.html`. V2.3 target:

- Waypoint editor accepts JSON arc definitions (already planned in V2.3
  configurable arc schema)
- Question groups are domain-specific and externalizable alongside
  vocabulary collections
- A clinical arc has different W1-W7 questions than a legal arc
- The editor allows a researcher to define questions per cultivar per
  waypoint without editing HTML

### Scene Editor → X3D Grid Coordinate Mapping

Currently affective zones are declared with positions and radii in the
scene editor, but those positions are not synchronized with the X3D scene
coordinate system. Required mapping:

- Scene editor coordinate space → X3D world coordinate space
- Zone radius → X3D geometry (sphere or cylinder marker in scene)
- Waypoint markers → X3D Transform positions
- Paths between zones → X3D route visualization

When a new scene is generated, zone positions from the field editor should
be baked into the X3D scene automatically. Keep the field layer and X3D
layer cleanly separated — the mapping is a one-way projection: field state
→ X3D scene. The X3D scene does not write back to the field (except via
ProximitySensor events through the existing /sensor endpoint).
